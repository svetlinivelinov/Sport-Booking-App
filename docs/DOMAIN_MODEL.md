# Domain Model

This document defines the sport-agnostic domain baseline for Sport Booking App v1.

## Core Principles

- Keep naming neutral (session, matchup, participant) instead of single-sport terms.
- Keep sport specifics in configurable rule modules.
- Share contracts across web and mobile through `packages/shared`.

## Entities

1. AppUser: identity, role, profile basics.
2. SportDefinition: sport metadata and rule strategy configuration.
3. Group: social/team container for participants.
4. Session: scheduled activity instance under a group and sport.
5. SessionParticipant: join table for users in a session.
6. Matchup: round/slot pairing generated by sport rules.
7. ScoreEntry: submitted score payload for a matchup.

## Relationship Overview

- One SportDefinition has many Groups and Sessions.
- One Group has many Sessions.
- One Session has many SessionParticipants and Matchups.
- One Matchup has many ScoreEntries.

## Rule Layer Contracts

- `generateMatchups`: scheduling strategy implementation.
- `validateScore`: scoring payload validation.
- `buildRanking`: leaderboard projection.

These are represented by `SportRules` in `packages/shared/src/domain/rules.ts`.
